포스트 마스터에게보고하는 데 어떤 문제가 있습니까?
notify_classes
사람을 가리키는 포스트 마스터 별명을 설정해야합니다. 사람들이 메일 배달 문제를보고 할 수 있도록이 별칭이 있어야합니다.

Postfix 시스템 자체도 문제를 포스트 마스터 별칭에보고합니다. 모든 유형의 문제 보고서에 관심이 없을 수 있으므로이보고 메커니즘을 구성 할 수 있습니다. 기본값은 심각한 문제 (자원, 소프트웨어) 만 포스트 마스터에게보고하는 것입니다.

기본 : resource, software

수업의 의미는 다음과 같습니다.

  • bounce: 배달 할 수없는 메일의 포스트 마스터 사본을 보냅니다.
  • 2 바운스 2bounce: 이중 바운스를 포스트 마스터에게 보냅니다.
  • delay: 지연된 메일을 포스트 마스터에게 알립니다. 이 경우 포스트 마스터는 메시지 헤더 만받습니다.
  • policy: (UCE) 정책 제한으로 인해 거부 된 클라이언트 요청을 포스트 마스터에게 알립니다. 포스트 마스터는 전체 SMTP 세션의 대화 내용을받습니다.
  • protocol: 포스트 마스터에게 프로토콜 오류 (클라이언트 또는 서버 측)를 알리거나 구현되지 않은 명령을 실행하려는 클라이언트의 시도. 포스트 마스터는 전체 SMTP 세션의 대화 내용을받습니다.
  • resource: 리소스 문제로 인해 배달되지 않은 메일을 포스트 마스터에게 알립니다 (예 : 큐 파일 쓰기 오류).
  • software: 소프트웨어 문제로 인해 배달되지 않은 메일을 포스트 마스터에게 알립니다.
