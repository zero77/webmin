Канонічне картографування
Необов'язковий канонічний файл вказує відображення адрес для локальних та не локальних адрес. Для відображення використовується демон очищення (8). Зображення адрес є рекурсивним.

Файл служить вводом до команди postmap (1). В результаті індексований файл у форматі dbm або db використовується для швидкого пошуку поштовою системою.

Канонічне відображення впливає на адреси заголовків повідомлень (тобто адреси, що відображаються всередині повідомлень), і адреси конвертів (наприклад, адреси, які використовуються в командах протоколу SMTP). Подумайте, набір правил Sendmail S3 , якщо ви хочете.

Як правило, можна використовувати канонічну таблицю для заміни імен для входу на ім'я Firstname.Lastname або для очищення адрес, створених застарілими поштовими системами.

Канонічне відображення не слід плутати з підтримкою віртуального домену. Використовуйте для цього віртуальну карту домену .

Канонічне відображення не слід плутати з місцевим псевдонімом. Для цього використовуйте карту псевдонімів пошти .

Формат канонічних відображень виглядає наступним чином: відображаються карти у порядку, як зазначено:

  • user@domain mapped to ... address :
    user@domain замінено address Ця форма має найвищий пріоритет. Ця форма корисна для очищення адрес, створених застарілими поштовими системами. Його також можна використовувати для створення адрес стилю Firstname.Lastname , але для більш простого рішення див. Нижче.
  • user відображений на ... address :
    user@site замінюється адресою, коли сайт дорівнює $myorigin , коли сайт вказаний у $mydestination або коли він вказаний у $inet_interfaces . Ця форма корисна для заміни імен для входу на ім'я Firstname.Lastname .
  • @domain відображено на ... address :
    Кожна адреса в домені замінюється адресою. Ця форма має найнижчий пріоритет.
У всіх вищезазначених формах, коли адреса має форму @otherdomain , результат є тим самим користувачем, що і в іншихдомену.

РОЗШИРЕННЯ АДРЕСИ: Коли пошук таблиці не вдається, а адреса localpart містить необов'язковий роздільник одержувача (наприклад, user+foo@domain ), пошук повторюється для необмеженої адреси (наприклад, user@domain ), і незрівняне розширення поширюється на результат пошуку таблиці. Порядок відповідності: user+foo@domain , user@domain , user+foo , user та @domain .
